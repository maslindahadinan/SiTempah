    36→3. Latar Belakang dan Pernyataan Masalah
    37→Pada masa ini, proses tempahan fasiliti gunasama di JTM seperti bilik mesyuarat, dewan, makmal komputer dan
    38→bilik seminar dijalankan secara manual melalui borang fizikal, e-mel atau panggilan telefon kepada kerani/pengurus
    39→fasiliti. Pendekatan ini menimbulkan beberapa isu operasi:
    40→(cid:127) (cid:127) Pertindihan tempahan (double booking) akibat ketiadaan paparan kekosongan fasiliti secara masa nyata
    41→(real-time).
    42→(cid:127) (cid:127) Proses kelulusan yang perlahan dan tidak telus kerana bergantung kepada komunikasi tidak berstruktur
    43→(e-mel/WhatsApp).
    44→(cid:127) (cid:127) Sukar untuk pemohon menyemak status permohonan tanpa perlu bertanya secara manual kepada pengurus.
    45→(cid:127) (cid:127) Ketiadaan rekod berpusat (centralised record) bagi tujuan audit, pelaporan dan analisis penggunaan fasiliti.
    46→(cid:127) (cid:127) Pengalaman pengguna (UX) borang sedia ada yang lapuk dan tidak mesra pengguna.
    47→Justeru, JTM bercadang membangunkan sebuah sistem web bersepadu — SiTempah — yang membolehkan

    49→(structured approval workflow) serta paparan status tempahan secara telus kepada semua pihak berkepentingan.

   196→9. Aliran Status Permohonan (Booking Status Workflow)
   197→Setiap permohonan tempahan akan melalui aliran status berikut. Reka bentuk status ini menyokong keperluan
   198→Pengurus untuk meluluskan, menangguhkan (KIV) atau menolak sesuatu permohonan.
   199→Status Penerangan Pencetus / Siapa Boleh
   200→Ubah
   201→Draf (Draft) Permohonan disimpan sementara sebelum dihantar rasmi. Pengguna
   202→Menunggu Kelulusan Permohonan telah dihantar dan menunggu tindakan Sistem (auto) selepas
   203→(Pending) Pengurus. penghantaran
   204→KIV (Kept-In-View) Permohonan ditangguhkan buat sementara — memerlukan Pengurus
   205→maklumat tambahan atau menunggu pengesahan lanjut.
   206→Diluluskan (Approved) Permohonan diluluskan; fasiliti disahkan ditempah untuk slot Pengurus
   207→berkenaan.
   208→Ditolak (Rejected) Permohonan ditolak berserta sebab/catatan penolakan. Pengurus
   209→Dibatalkan (Cancelled) Permohonan dibatalkan oleh pemohon sendiri sebelum tarikh Pengguna / Pentadbir
   210→acara.
   211→Tamat Tempoh (Expired) Permohonan automatik ditamatkan jika tiada tindakan Sistem (auto)
   212→Pengurus selepas tempoh SLA ditetapkan.
   213→Rajah Aliran Status (Ringkasan Teks)
   214→Draf fi Menunggu Kelulusan fi { Diluluskan | KIV fi (Diluluskan | Ditolak) | Ditolak } ; Menunggu Kelulusan / KIV fi
   215→Tamat Tempoh (jika SLA terlampau) ; Diluluskan fi Dibatalkan (atas permintaan pemohon).

   279→GLM 5.2
   280→Backend-as-a-Servi Supabase (PostgreSQL, Supabase Auth, Pangkalan data, pengesahan pengguna, storan fail
   281→ce Supabase Storage, Realtime lampiran, dan kemas kini masa nyata (status
   282→Subscriptions, Row Level Security) tempahan).
   283→Logik Sisi Pelayan Supabase Edge Functions / Netlify Logik perniagaan kompleks: semakan konflik
   284→(Backend Logic) Functions (Node.js/TypeScript) tempahan, penghantaran notifikasi e-mel,
   285→penjanaan laporan.
   286→Pengesahan Supabase Auth (E-mel & Kata Laluan / Log masuk selamat, pengurusan sesi (JWT), dan
   287→Pengguna (Auth) Magic Link) integrasi RBAC melalui custom claims/roles table.
   288→Notifikasi Supabase Edge Functions + Menghantar e-mel automatik apabila status

   405→Antara muka SiTempah akan menggunakan gaya reka bentuk Glassmorphism — panel separa lut sinar
